## Plan: Var-Only VM

Remove `Value` as a distinct runtime concept and make `Var` the only semantic unit across compilation and execution. Globals, locals, literals, and temporaries all become `Var`s. The compiler discovers global `Var` declarations directly from source, lowers literals into constant `Var`s in a BSS/constant area, allocates locals and intermediates as `Var`s, and the evaluator operates on a stack of `*Var`. The recommended implementation is still staged, but it now commits to `Var`-only execution rather than a mixed `Var` metadata plus `Value` cell design.

**Steps**
1. Phase 1: Define the canonical `Var` model in the VM package. `Var` must carry `Index`, `Type`, `Flags`, and `Value`, and the implementation must define what `Value` contains for scalar, string, bool, const, pointer-backed, local, and temporary variables. This blocks all later steps.
2. Phase 1: Decide and document identity/allocation rules for every `Var` class: user-declared globals, compiler-emitted literal constants, compiler-emitted locals, and compiler-emitted temporaries. The simplest coherent rule is one flat index space or one index space per storage segment with an explicit storage-class tag. This blocks bytecode and memory layout.
3. Phase 1: Replace `CompilerGlobalStateInterface.GetGlobalVarID(name)` with declaration discovery from parsed package-level `Var` definitions. The compiler should synthesize global metadata directly from user-authored declarations and stop depending on a free-standing `map[string]ID`.
4. Phase 1: Replace placeholder globals in the UI package with user-declared `Var` values and make those declarations authoritative for index, type, flags, and initial value. Preserve the current source-style goal where bare identifiers like `UiMode`, `Finger0State`, and `GroundfloorBathroomMirrorLight_State` remain valid in page code.
5. Phase 2: Remove `Value` from the runtime model in [vm/value.go](/Users/obnosis5/dev.go/src/github.com/jurgen-kluft/go-esp32-ui/vm/value.go) and replace it with `Var`-centric helpers. Any conversion, coercion, comparison, arithmetic, or load/store helper should operate on `Var` or `*Var` directly.
6. Phase 2: Redesign bytecode operands and VM memory layout around `Var` references instead of packed primitive values. The current packed `ID {Type:8, Idx:24}` encoding can survive only if it is reinterpreted as a compact `Var` reference into VM-managed storage; otherwise the operand format must change. This blocks execution changes.
7. Phase 2: Introduce a VM-managed storage model for all `Var`s. Globals, literals in BSS/constant storage, locals, and temporaries must all live in addressable `Var` slots so the evaluator can push `*Var` onto its stack. Define whether temporaries are recycled stack-frame slots or entries in a dedicated temp arena.
8. Phase 2: Extend compiler expression lowering so identifier resolution is explicitly three-way: literals, declared globals, and locals. Literals lower to compiler-emitted constant `Var`s. Declared globals come from parsed package-level `Var` declarations. Locals are compiler-emitted `Var`s allocated per block/frame. Preserve local-shadowing-global behavior.
9. Phase 2: Rework assignment and load/store generation so expressions pass around `*Var` and write through to referenced `Var` storage. This includes global loads, local loads, local assignment targets, and temporary/intermediate results. This depends on steps 6 and 7.
10. Phase 3: Define expression semantics over `Var`s. Arithmetic, comparisons, boolean tests, unary operators, and syscall argument passing must all operate on `Var` operands, including compiler-emitted constant vars for literals. The recommended rule is: operators read two operand `Var`s where applicable, produce a temporary result `Var`, and push a pointer to that temporary. Under this rule, `Finger0State == 0` is a comparison between the `Finger0State` var and a constant-literal var for `0`.
11. Phase 3: Implement literals as constant `Var`s in BSS or a constant segment. Integer, float, bool, and string literals should all become immutable `Var`s, with `VarFlagConst` preventing mutation. String literals and const ASCII strings should specifically become `VarTypeStr` vars with `VarFlagPtr`, where `Value` points at the string data stored in the binary. This directly matches your requirement that used literals become `Var`s.
12. Phase 3: Define pointer-backed vars (`VarFlagPtr`). A pointer-backed `Var` should keep its external storage link in `Value`, with read/write helpers dereferencing through a controlled adapter path. This includes string vars whose `Value` points at binary-resident ASCII text. Do not let arbitrary Go pointers leak into general operator logic without a narrow access contract.
13. Phase 3: Rework VM syscall dispatch and text rendering around `Var` references. `DrawText` should take the text operand as a `Var`, allowing it to render either a constant string var or another string-producing var directly. Under this model, `DrawVar` becomes redundant and should be removed once all call sites are migrated to `DrawText(..., text Var)` semantics.
14. Phase 4: Revisit the page-language rule for bare identifiers. If source syntax stays unchanged, the compiler must lower the same bare `Var` identifier differently by context: as a value-bearing operand in `Finger0State == 0`, as an assignment target in `UiMode = ModeStandardGrid`, and as a variable reference in calls such as `DrawText(..., DateString)` where the callee needs the string var itself. This rule must be explicit and tested.
15. Phase 4: Update tests in `vm` to reflect a `Var`-only evaluator, add compilation tests for global/local/literal classification, add execution tests for pointer-to-`Var` stack behavior, and add regressions for current page patterns such as `if UiMode == ModeStandardGrid`, `if Finger0State == 0`, `DrawText(..., DateString)`, and `SetLightBrightness(..., uint32((Finger1X-90)*100/300))`.
16. Phase 4: Clean up the inconsistent `NewCompiler` constructor and global plumbing so the compiler has one supported declaration-discovery path and one supported runtime `Var` storage path.

**Decisions**
- Included: removing `Value` as a separate concept, using `Var` for globals, locals, literals, and temporaries, operating the evaluator on a stack of `*Var`, and preserving current page authoring style when feasible.
- Excluded from the first implementation slice: broad page API redesign unless the preserved syntax proves too costly, and unrestricted raw pointer behavior without a narrow `VarFlagPtr` access contract.
- Confirmed architectural rule: expressions operate on `Var`s, not on primitive `Value` cells. Intermediate results are temporary `Var`s, and comparisons such as `Finger0State == 0` are always `Var`-to-`Var` operations because the literal side is lowered to a constant var.
- Confirmed constant rule: used literals become constant `Var`s in BSS or a constant segment rather than remaining inline primitive values. String and ASCII text constants specifically become `VarTypeStr | VarFlagPtr` vars that point at binary-resident string data.
- Confirmed UI text rule: `DrawText` should accept the text operand as a `Var`, which makes `DrawVar` redundant and a candidate for removal after call-site migration.
- Recommended implementation caution: even with `Var`-only semantics, define a small set of accessor/coercion helpers so arithmetic, comparisons, and string dereference logic do not spread across the VM.

**Further Considerations**
1. The hardest semantic edge remains context-sensitive use of a bare declared `Var`: the same identifier is used today as a value (`Finger0State == 0`), as an assignment target (`UiMode = ModeStandardGrid`), and as a variable reference (`DrawText(..., DateString)`). The compiler rule for this must be explicit.
2. `Var.Value interface{}` is coherent with your design, but it will make operator performance and debugging depend on strict helper functions for numeric, bool, and string coercion. That is workable, but it should be centralized.
3. The current `NewCompiler` API is already inconsistent across files and tests. Fold that cleanup into the declaration-discovery and `Var`-storage migration so the compiler has a single supported constructor shape.
